自定义带删除功能的EditText

发表于 2016-04-26   |  

先放效果图:

提出问题

如图所示,是一个我们现在应用中经常使用的一个控件,一个带删除功能的EditText,可能在你自己的app中所有的EditText控件都带删除功能,如果我们笨笨的使用线性或者相对布局来排列这三个控件,这样会出现两个问题:

  1. 代码过长,不易于查看,想想如果一个界面里面出现3个以上这种控件,xml布局中会出现很冗长的重复代码;

  2. 同样不易于维护,当产品要对整个app的EditText做出UI的修改的时候,你会发现很痛苦的在一个一个xml布局的修改,因为一个app要保证风格的统一,所有的对应的控件都需要修改;

    所以当我们发现一个app的很多界面有相同的控件时,我们就要留心将他们封装成一个自定义的控件,在使用的时候我们就复用这个控件就可以了.

代码实现

自定义控件我们通常分为两种:自定义组合控件和自定义全新的控件,而我这个控件是用自定义组合控件来实现的:

  1. 定义属性

    <declare-styleable name="CleanEditText">
    <attr name="leftImg" format="reference" />
    <attr name="rightImg" format="reference" />
    <attr name="hint" format="string|reference" />
    <attr name="text" format="string|reference" />
    <attr name="textColor" format="color|reference" />
    <attr name="textHintColor" format="color|reference"/>
</declare-styleable>

  2. 定义自定义控件的布局

  3. 定义自定义控件

    public CleanEditText(Context context, AttributeSet attrs){

    View view = LayoutInflater.from(context).inflate(R.layout.weight_clear_edittext, this, true);
    leftImg = (ImageView) view.findViewById(R.id.iv_left);
    rightImg = (ImageView) view.findViewById(R.id.iv_right);
    et_text = (EditText) view.findViewById(R.id.et_text);
    et_text.addTextChangedListener(textWatcher);
    rightImg.setVisibility(View.INVISIBLE);

    rightImg.setOnClickListener(new OnClickListener() {
      @Override
      public void onClick(View v) {
          et_text.setText("");
      }
    });
    TypedArray ta = context.obtainStyledAttributes(attrs, R.styleable.CleanEditText);

    设置属性(将属性通过ta.getString(),getColor()等方法获取自定义的属性值)

    ta.recycle();
}

  4. 对edittext的监听(删除功能)

    //输入框文本变化监听器
private TextWatcher textWatcher = new TextWatcher() {
    @Override
    public void beforeTextChanged(CharSequence s, int start, int count, int after) {
    }

    @Override
    public void afterTextChanged(Editable s) {
    }

    @Override
    public void onTextChanged(CharSequence s, int start, int before, int count) {
        if (TextUtils.isEmpty(s)) {
            rightImg.setVisibility(View.INVISIBLE);
        } else {
            rightImg.setVisibility(View.VISIBLE);
        }
    }
};

  5. 对外提供各种修改的set,get方法(getEditText()方法将et暴露出去可以提供各种其他的客制化,如设置inputType属性等)

    protected void setHint(String hint) {
    et_text.setHint(hint);
}

protected void setText(String text) {
    et_text.setText(text);
}

protected String getText() {
    return et_text.getText().toString();
}

protected EditText getEditText() {
    return et_text;
}

  6. 定制化的行为,点击左边图片有特定的事件,所以我们在不同的界面中实现的功能是不一样的,这样我们就不能在其中直接添加具体的实现逻辑,通过接口回调的方式,将具体的实现交由调用者自己事件(没有在工程中实现)

    //在控件中定义一个接口,创建方法
public interface CleanEditTextClickListener{
    //左边图片的点击事件
    void leftClick();
}

//暴露方法,交由调用者调用来实现接口回调
public void setOnClickerListener(CleanEditTextClickListener mListener){
    this.mListener = mListener
}

//在控件中添加点击事件,但是不要去实现,而是调用接口中相应的方法
leftImg.setOnClickListener(new OnClickListener(){
    @Override
    public void onClick(View v){
        mListener.leftClick();
    }
});

//调用者实现接口回调
mCleanEditText.setOnClickerListener(
    new CleanEditTextClickListener(){
        @Override
        public void leftClick(){
            //定义事件
        }
    }
);

  7. 交由调用者来使用,在xml布局中

    <com.jinjin.clearedittext.CleanEditText
    xmlns:app="http://schemas.android.com/apk/res-auto"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"
    app:hint="请输入用户名"
    app:leftImg="@drawable/icon_login_user"
    app:rightImg="@drawable/icon_delete"
    app:text="输入的text"
    app:textColor="#ff0000"/>

写在最后

这只是一个最简单的自定义组合控件,但是在复杂的组合控件都是由以上这些步骤实现的:

定义属性->组合控件->定义接口->调用者使用.

Android系统控件与自定义控件(二)

发表于 2016-04-26   |  

前文提到android系统控件的机制,我们可以通过继承系统组件,方便的拓展功能,甚至可以直接自定义一个控件,实现Android系统中没有的功能.
适当的使用自定义View,可以大大提高我们的开发效率和应用的体验效果,但是滥用自定义View则会带来适得其反的效果.

View中比较重要的回调方法

  • onFinishInflate():从布局文件加载后回调;
  • onSizeChanged():当组件大小发生改变的时候回调;
  • onMeasure():回调进行测量;
  • onLayout():回调确认显示的位置;
  • onTouchEvent():监听触摸事件;

对现有控件进行拓展

通过继承现有的系统控件,对其进行拓展,增加新的功能,修改显示的效果.

自定义组合控件

当一个app中需要使用到大量的相同形式的UI界面的时候,通常可以使用这种控件.

通过继承一个适当的ViewGroup,在ViewGroup中添加子View的形式组合成一个新的组合控件.通常我们会通过自定义属性的方式给这个组合控件更高的扩展性.

Android系统控件与自定义控件(一)

发表于 2016-04-26   |  

控件是每个app必不可少的一部分,无论是Android的系统控件,还是我们自定义的控件;而这些控件,组成了我们app的精美的界面.

Android控件

  1. Andoird中的每个控件都是矩形.

  2. 在Activity的onCreate()方法中我们都会使用setContentView()的方法来加载我们的布局文件,这样我们的布局内容才显示出来;首先我们来看看Android界面的架构图:

    如上图所示,每个Activity中都包含一个PhoneWindow对象,PhoneWindow对象又将一个DecorView设置为应用窗口的根View;DecorView对象作为一个窗口界面的顶层视图,封装了窗口的通用方法;DecorView又将窗口分为了两部分,一个是TitleView,一个是ContentView;这个ContentView就是我们所熟悉的;它是一个名为content的FrameLayout,而我们的布局文件就是设置在这个FrameLayout中.

View的绘制原理

Android在绘制View的时候,遵循我们日常生活中的规律:测量,绘制

测量

  • 在现实生活中,我们在绘制一个东西的时候,首先我们需要测量东西的大小,而在Android中我们在onMeasure()方法中执行;
  • Android给我们提供了MeasureSpec来实现我们的测量;
    • EXACTLY: 精确值. 当指定为具体数值或者match_parent;
    • AT_MOST: 最大值. 当指定为wrap_content;
@Override
protected void onMeasure(int widthMeasureSpec, int heightMeasureSpec) {
    setMeasuredDimension(measureWidth(widthMeasureSpec), measureHeight(heightMeasureSpec));
}

private int measureWidth(int measureSpec) {
    int result = 0;
    int specMode = MeasureSpec.getMode(measureSpec);
    int specSize = MeasureSpec.getSize(measureSpec);
    if (specMode == MeasureSpec.EXACTLY) {
        result = specSize;
    } else {
        result = 200; //当为wrap_parent时指定默认的宽
        if (specMode == MeasureSpec.AT_MOST) {
            result = Math.min(result, specSize);
        }
    }
    return result;
}

private int measureHeight(int measureSpec) {
    int result = 0;
    int specMode = MeasureSpec.getMode(measureSpec);
    int specSize = MeasureSpec.getSize(measureSpec);
    if (specMode == MeasureSpec.EXACTLY) {
        result = specSize;
    } else {
        result = 200; 
        if (specMode == MeasureSpec.AT_MOST) {
            result = Math.min(result, specSize);
        }
    }
    return result;
}

测量(用于ViewGroup)

  • ViewGroup在测量时通过遍历所有的子view,调用子view的Measure方法获得每个子view的测量结果;
  • 之后遍历来调用子view的Layout方法,并指定其具体的位置,从而决定布局位置;
  • 注意:如果需要支持wrap_parent的话,同样需要重写onMeasure()方法;

绘制

  • 当我们测量好了之后就可以进行绘制了,Android中提供了onDraw()方法和Canvas来进行绘制;

    @Override
protected void onDraw(Canvas canvas) {
    //绘制外层矩形
    canvas.drawRect(0, 0, getMeasuredWidth(), getMeasuredHeight(), mPaint1);
    //绘制内层矩形
    canvas.drawRect(0, 0, getMeasuredWidth() - 10, getMeasuredHeight() - 10, mPaint2);
    //用来保存Canvas的状态。save之后，可以调用Canvas的平移、放缩、旋转、错切、裁剪等操作
    canvas.save();
    canvas.translate(10, 0);
    //textview 绘制文字
    super.onDraw(canvas);
    //用来恢复Canvas之前保存的状态。防止save后对Canvas执行的操作对后续的绘制有影响
    canvas.restore();
}
  • 而ViewGroup通常不需要绘制,因为没有要绘制的内容.它会调用dispatchDraw()来绘制子view.

Android的生命周期和启动模式

发表于 2016-04-25   |  

Android正常情况

activity第一次启动:

onCreate->onStart->onResume

打开新的activity或者返回桌面时

1. onPause->onStop
2. 新的activity为透明主图 onPause

返回activity时

onRestart->onStart->onResume

back键

onPause->onStop->onDestroy

Android异常情况

当资源相关的配置信息发生改变或者系统内存不足时,activity可能被杀死.

   onSaveInstanceState(将当前状态保存到Bundle对象中) 
-> onStop() 
-> onDestory() 
-> onCreate(保存的Bundle对象) 
-> onStart() 
-> onRestoreInstanceState(保存的Bundle对象)

我们可以通过上面的机制,在被销毁的时候onSaveInstanceState()方法中存入我们想要保存的数据,在onCreate和onRestoreInstanceState中将保存起来的数据再显示到activity中.

我们可以通过给activity指定configChanges属性来使activity不重新创建:

Activity的启动模式

activity的LaunchMode

  1. standard:标准模式,系统默认的模式

    • 每次启动一个activity都会重新创建一个新的实例,不管这个实例是否已经存在;
    • 谁启动了activityA,那么A就在启动他的activity的任务栈中;
    • 注意:由于非activity类型的Context(如ApplicationContext)并没有任务栈,这个时候由他来启动standard模式的activity的时候会报错;
    • 我们可以通过为待启动的activity指定 FLAG_ACTIVITY_NEW_TASK 标记,这个时候这个新的activity就会在一个新的任务栈中,实际上就是singleTask模式启动的;

  2. singleTop:栈顶复用模式

    • 当新的activity已经处于栈顶的时候,就不会创建新的实例;这个时候会调用 onNewIntent ,通过这个方法,我们可以取出当前请求的信息;
    • singleTop模式: ABCD -> 启动D -> ABCD
    • standard模式: ABCD -> 启动D -> ABCDD

  3. singleTask:栈内复用模式

    • 单实例模式:当activity在任务栈中存在时,那么启动activity不会创建新的实例;也会调用 onNewIntent ;
    • ABCD -> 启动B -> AB,会将CD出栈;

  4. singleInstance:单实例模式

    • 这种模式下的activity智能单独存在于一个任务栈中;
    • AB:任务栈1 -> 启动C -> AB:任务栈1;C:任务栈2;

给Activity指定启动模式

  1. 在AndroidMenifest.xml中指定:

    <activity
android:name=".home.HomeActivity"
android:launchMode="singleTask"
android:screenOrientation="portrait"/>

  2. 在Intent中设置标记位指定启动模式:

    Intent intent = new Intent(this,FeedbackActivity.class);
intent.addFlags(intent.FLAG_ACTIVITY_NEW_TASK);
startActivity(intent);

  3. 第二种的优先级大于第一种的优先级;同时存在时,以第二种为准;

Activity的Flags标记位

  1. FLAG_ACTIVITY_NEW_TASK

    指定singleTask启动模式,和ActivityManifest.xml相同;

  2. FLAG_ACTIVITY_SINGLE_TOP

    指定singleTop启动模式,和ActivityManifest.xml相同;

  3. FLAG_ACTIVITY_CLEAR_TOP

    当有此标记位的activity启动的时候,在同一个任务栈中的在它上面的activity都会出栈,通常与FLAG_ACTIVITY_NEW_TASK配合使用,这种情况下,被启动的activity如果已经存在,则调用onNewIntent方法;如果是standard模式,被调用的activity也会出栈,系统会重新创建activity实例放入栈顶;

  4. FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS

    有此标记位的activity不会显示在最近打开的activity列表里,等同于android:excludeFromRecents="true";

intentFilter匹配规则

  1. 匹配规则

    1. 隐式调用在使用的时候需要intent匹配intentFilter所设置的过滤信息,如果不匹配的话则不能启动activity;
    2. 一个activity可以有多个intent-filter,一个intent-filter中可以有多个action,data,category,而一个intent只要匹配一个intent-filter就可以启动activity;

  2. action

    action是一个字符串,系统有预定义的action,我们也可以自定义action;
    匹配规则: 一个intent-filter可能有多个action,只要intent匹配其中一个就满足;如果intent没有指定,则匹配失败;

  3. category

    category是一个字符串,系统有预定义的category,我们也可以自定义category;
    匹配规则: 一个intent-filter可能有多个category,而intent也只要匹配一个就可以了,也可以不匹配;因为系统在调用startActivity或者startActivityForResult的时候会默认加上android.intent.category.DEFAULT这个category;所以我们应该在intent-filter中添加android.intent.category.DEFAULT这个category;

  4. data

    和action类似;
    data由mimeType,URI组成:

    1. mimeType指类型,如image/jpeg等;
    2. URI: <scheme>://<host>:<post>/[path]|[pathPrefix]|[PathPattern]

  5. 注意:当我们通过隐式调用一个activity的时候,可以做一下判断,看是否有activity匹配intent:intent的resolveActivity方法或者PackageManager的resolveActivity方法,这两个方法在找不到activity的时候会返回null;
